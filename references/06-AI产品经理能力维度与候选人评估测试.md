# AI 产品经理能力维度与候选人评估测试

## 0. 使用场景

用于评估 AI 产品经理候选人，尤其适合判断非技术背景、设计/内容/运营/行业背景转 AI 产品的人，是否已经具备进入 AI 产品岗位的基本能力。

核心判断不是“会不会写代码”，而是：

```text
能否发现真实问题 -> 设计 AI 产品方案 -> 做出可验证原型 -> 定义评估指标 -> 推动上线与迭代 -> 控制风险和成本
```

## 1. 能力维度总览

| 维度 | 权重 | 主要看什么 | 合格线 |
|---|---:|---|---|
| 产品基本功 | 15 | 用户、需求、流程、PRD、原型、优先级 | 能独立拆需求和画流程 |
| AI 技术认知 | 15 | LLM、Prompt、RAG、Agent、API、幻觉、上下文、成本 | 能和研发讨论方案边界 |
| 数据与评估 | 12 | 指标、测试集、人工评测、A/B、采纳率、准确率 | 能定义效果是否变好 |
| Vibe coding 与原型 | 12 | Cursor/Bolt/Lovable/Coze/Dify、Demo、API 调用 | 能 1-3 天做可演示原型 |
| 业务与场景判断 | 12 | 业务痛点、用户流程、付费价值、行业理解 | 知道 AI 为什么值得做 |
| 项目经验与落地 | 12 | 从想法到交付、协作、复盘、结果 | 有项目证据，不只会讲概念 |
| 协作推进 | 8 | 研发、算法、设计、运营、销售、客户协作 | 能把模糊需求推进成任务 |
| 风险与责任意识 | 8 | 隐私、安全、版权、偏见、内容安全、人工兜底 | 知道 AI 不能无审查上线 |
| 表达与文档 | 6 | PRD、面试表达、方案汇报、结构化沟通 | 能讲清楚问题、方案、结果 |
| 总分 | 100 |  | 70 分以上可进入业务面 |

## 2. AI 产品经理需要达到的编程理解程度

| 内容 | 非编程专业了解程度 | AI 产品经理了解程度 |
|---|---|---|
| HTML / CSS / JavaScript | 知道网页由结构、样式、交互组成 | 能判断前端交互是否合理，能改简单文案和页面 |
| TypeScript | 知道是更稳的 JavaScript | 知道中大型前端常用，理解类型检查能减少错误 |
| Python | 知道常用于 AI、数据、自动化 | 能读懂简单脚本，能跑 Notebook 或 API Demo |
| Jupyter Notebook | 知道是实验文档 | 能运行、看图表、理解模型/数据实验结果 |
| Shell / PowerShell | 知道是命令行自动化 | 能看懂启动、安装、报错等基础命令 |
| Java | 知道常用于后端和企业系统 | 知道 AI 能力如何接入现有后端系统 |
| TeX | 知道是论文/公式排版 | 非核心，除非做学术工具 |

AI PM 的技术合格线：

```text
不要求写生产级代码，但要知道功能在哪一层实现、数据怎么流、模型怎么接入、成本和风险在哪里。
```

## 3. Vibe coding 能力等级

| 等级 | 描述 | 对 AI PM 的要求 |
|---|---|---|
| L0 | 只会让 AI 写代码，不会运行，不会看报错 | 不合格 |
| L1 | 能用 AI 生成简单页面或小工具 | 入门 |
| L2 | 能独立做可演示原型：页面、流程、假数据、简单交互、API 调用 | AI PM 合格线 |
| L3 | 能做小型 MVP：前端 + 简单后端 + 数据保存 + AI 调用 | 强加分 |
| L4 | 能负责生产级代码、架构、安全、性能、部署 | 工程师要求，AI PM 不强求 |

候选人至少应达到 L2：

- 能把产品想法写成清晰 Prompt。
- 能让 AI 生成原型并跑起来。
- 能修改页面文案、表单、按钮、流程。
- 能接一个简单模型 API 或无代码 AI 工作流。
- 能识别 Demo 哪些是真功能，哪些是假数据。
- 能把原型转化成研发可接手的需求。

## 4. 职责与工作目标

### 4.1 主要职责

1. 识别适合 AI 介入的业务场景。
2. 定义目标用户、核心问题、输入输出和业务流程。
3. 设计 AI 产品形态：助手、Copilot、Agent、RAG 问答、生成工具、工作流自动化等。
4. 编写 PRD、原型、流程图和验收标准。
5. 与研发/算法确认模型、数据、API、RAG、Agent、成本和延迟方案。
6. 设计 Prompt、知识库、评测集和反馈闭环。
7. 定义产品指标和模型效果指标。
8. 推动灰度、上线、用户反馈和版本迭代。
9. 管理隐私、安全、幻觉、版权、内容安全和人工兜底。

### 4.2 工作目标

| 目标类型 | 示例 |
|---|---|
| 效率目标 | 节省用户 30% 写作、检索、处理时间 |
| 质量目标 | 生成内容采纳率提升，错误率下降 |
| 业务目标 | 提升转化率、留存、付费率、续费率 |
| 成本目标 | 降低客服、审核、运营、交付成本 |
| 体验目标 | 减少操作步骤，提高任务完成率 |
| 风险目标 | 降低幻觉、违规输出、投诉和误用 |
| 数据目标 | 建立可持续优化的数据反馈闭环 |

## 5. 候选人分级标准

| 等级 | 分数 | 判断 |
|---|---:|---|
| S | 85-100 | 可独立负责 AI 产品模块，有真实项目、评估体系和落地经验 |
| A | 75-84 | 能胜任初中级 AI PM，能做原型、写需求、推动交付 |
| B | 65-74 | 有潜力，但 AI 技术/评估/项目证据需补强 |
| C | 50-64 | 更适合 AI 产品运营、助理 PM、Workflow 配置类岗位 |
| D | 0-49 | 暂不建议进入 AI PM 岗位，应先补项目和基础认知 |

## 6. 面试评估题库

### 6.1 AI 产品理解

1. 你如何理解 AI 产品经理和普通产品经理的区别？
2. 什么场景适合用 AI，什么场景不适合？
3. Chatbot、Copilot、Agent、Workflow 有什么区别？
4. 如果模型经常幻觉，你会怎么设计产品兜底？
5. RAG 和微调分别适合什么情况？

### 6.2 产品设计

1. 设计一个 AI 简历分析助手。
2. 设计一个 AI 客服助手。
3. 设计一个面向设计师的 AIGC 创意工具。
4. 你会如何定义 MVP？
5. 你会如何设计用户反馈机制？

### 6.3 数据与评估

1. 你如何判断一个 AI 功能效果变好了？
2. 生成质量怎么评估？
3. 如何设计人工评测表？
4. 准确率、召回率、采纳率、满意度分别适合什么场景？
5. 如果用户说“感觉不好用”，你怎么定位问题？

### 6.4 Vibe coding

1. 你用 AI coding 工具做过什么 Demo？
2. 遇到代码报错时你怎么处理？
3. 原型和生产级产品的区别是什么？
4. 如何把一个 Vibe coding Demo 转成研发需求？
5. 你如何判断 AI 生成的代码是否可以上线？

### 6.5 项目深挖

1. 讲一个你做过的 AI 项目。
2. 这个项目为什么必须用 AI？
3. 你的具体职责是什么？
4. 项目如何验证有效？
5. 如果重做一次，你会改哪里？

## 7. 实操测试设计

### 测试 A：AI 产品方案题，60 分钟

题目：

```text
为求职者设计一个 AI 简历与 JD 匹配助手，帮助用户判断岗位是否适合投递，并给出简历修改建议。
```

交付物：

1. 目标用户与核心场景。
2. 用户流程图。
3. MVP 功能列表。
4. 输入、处理、输出说明。
5. 模型/RAG/Prompt/API 使用方案。
6. 效果评估指标。
7. 风险与兜底方案。

评分：

| 项目 | 分值 |
|---|---:|
| 问题定义清晰 | 15 |
| 流程完整 | 15 |
| AI 方案合理 | 20 |
| 指标可评估 | 20 |
| 风险意识 | 15 |
| 表达清楚 | 15 |

### 测试 B：Vibe coding 原型题，2-4 小时

题目：

```text
用任意 AI coding 工具做一个可演示原型：输入 JD 和简历摘要，输出匹配度、风险点和修改建议。
```

最低要求：

- 有输入框。
- 有结果展示。
- 有 3 类输出：匹配点、风险点、修改建议。
- 可以用假数据或模拟 API，但必须标注。
- 能本地运行或在线演示。

加分项：

- 接入真实模型 API。
- 支持导出 Markdown。
- 有加载、错误、空状态。
- 有反馈按钮。
- 有清楚 README。

评分：

| 项目 | 分值 |
|---|---:|
| 能跑起来 | 20 |
| 产品流程完整 | 20 |
| AI 输出结构合理 | 20 |
| 交互和信息层级清楚 | 15 |
| 能说明技术边界 | 15 |
| 文档清楚 | 10 |

### 测试 C：评估与复盘题，30 分钟

题目：

```text
一个 AI 总结功能上线后，用户反馈“有时不准、有时废话多”。你如何定位并改进？
```

优秀回答应覆盖：

- 问题分类：事实错误、遗漏重点、表达冗长、格式不稳定、用户预期不一致。
- 数据收集：样本、用户反馈、失败案例、日志。
- 评估方式：人工评分表、测试集、采纳率、重试率、投诉率。
- 改进方案：Prompt、RAG、结构化输出、模型切换、交互确认、人工兜底。
- 上线策略：灰度、A/B、回滚、监控。

## 9. 红旗信号

- 只会说“AI 很火”，讲不清具体场景。
- 把 Prompt 当成全部 AI 产品能力。
- 没有指标意识，无法判断效果好坏。
- 不知道幻觉、隐私、版权、内容安全风险。
- Vibe coding 只会生成，不能运行、修改或解释。
- 项目只展示作品，没有用户、问题、方案、结果。
- 把 Demo 当成生产级产品，不知道工程、数据和安全边界。

## 10. 录用建议

| 岗位类型 | 推荐标准 |
|---|---|
| AI 产品实习/助理 | 60 分以上，Vibe coding L1-L2，有学习速度 |
| 初级 AI 产品经理 | 70 分以上，Vibe coding L2，有完整 Demo 或项目 |
| 中级 AI 产品经理 | 80 分以上，能定义指标、评估体系和上线策略 |
| AI Workflow / Agent 配置岗 | 65 分以上，强原型和流程拆解能力 |
| 高级 AI 平台/算法产品 | 85 分以上，并需额外考察模型、数据、平台和工程协作深度 |

最小合格标准：

```text
70 分以上 + Vibe coding L2 + 至少 1 个 AI 产品 Demo/项目 + 能讲清评估指标和风险兜底。
```
